# RadClean 平台功能与 AI 融合总结

## 一、平台功能概述

RadClean 面向放射科 CT 数据的多层级治理，按"净化 → 结构化 → 验证"三阶段流水线设计。每个阶段解决数据治理中的一个核心问题。

---

## 二、各阶段功能与底层逻辑

### Stage 1 — 数据净化

**做什么：** 对 DICOM 文件包进行文件级和字段级清洗。

**底层逻辑：**
- **文件层：** 扫描数据包，识别并剔除临时文件、日志文件等非影像残留；统一文件命名规范（如检查号对齐）；校验每个病例的影像文件完整性（是否缺层、是否损坏）
- **字段层：** 读取 DICOM 元数据字段，按预定义规则标准化——年龄从混合格式（"077Y"、\"77岁\"）统一为整数；性别从多语言缩写（M/F/Male/Female/男/女）统一为标准值；厂商名称做别名映射（如\"GE MEDICAL SYSTEMS\"→\"GE\"）；HU 参数（RescaleIntercept、RescaleSlope）清洗异常值

**传统方式：** 人工逐份核对 DICOM 头信息，用 Excel 记录异常。100 个病例约需 2-3 个工作日，且容易遗漏不一致写法（如同一厂商有 5 种拼写）。

**我们的 AI 方式：** 规则引擎自动批量处理，30+ 条预定义清洗规则同时运行。确定性规则保证 100% 可复现。120 个病例约 2 秒完成。

| 维度 | 传统方式 | RadClean |
|------|---------|----------|
| 处理方式 | 人工逐份核对 | 规则引擎批量自动化 |
| 耗时（120 例） | 2-3 工作日 | ~2 秒 |
| 一致性 | 依赖审核者经验 | 确定性规则，100% 可复现 |
| 覆盖面 | 主要字段 | 文件+字段全维度 |

---

### Stage 2 — 报告结构化

**做什么：** 将放射科自由文本报告转化为结构化 JSON。

**底层逻辑：**
- **结构层：** 识别报告段落（临床诊断、影像所见、诊断意见），拆分为可独立处理的信息单元
- **语义层：** 基于领域词典（21 类疾病、50+ 解剖部位、30+ 修饰词）进行实体识别；句子级否定检测区分"存在"与\"排除"（如\"未见结节\"→结节被否定）；关联疾病与解剖部位形成结构化三元组
- **增强层（可选 LLM）：** 对规则引擎结果进行 13 字段深度提取（器官、病变、测量值、分布、侧面等），补充规则引擎覆盖不到的细粒度信息

**传统方式：** 研究人员逐份阅读报告，手动填入 Excel 模板。因不同医生书写习惯差异大（同一种疾病十余种表述），工作量大且标注一致性低。多中心数据更是噩梦。

**我们的 AI 方式：** 规则引擎先行，词典 + 否定检测完成确定性标注（覆盖常规表述）；LLM 作为增强模块处理复杂措辞和长尾表述。规则引擎不依赖 LLM 也可独立运行。

| 维度 | 传统方式 | RadClean |
|------|---------|----------|
| 处理方式 | 人工阅读+手动填表 | 词典匹配 + 否定检测 + 可选 LLM |
| 标注一致性 | ~60-70%（不同标注者） | 规则部分 100%，LLM 部分 ~85-95% |
| 语义深度 | 通常仅疾病名 | 21 疾病 + 50+ 部位 + 30+ 修饰词 |
| 可扩展性 | 每加一个病种需重新培训 | 词典加一行即可 |

---

### Stage 3 — 治理验证

**做什么：** 验证治理结果的质量，并给出量化评分。

**底层逻辑：**
- **完整性：** 检查 5 个核心 DICOM 字段（PatientName、PatientID、StudyDate、Modality、AccessionNumber）的缺失率，阈值 ≤ 15%
- **准确性：** 检测年龄范围（0-120）、HU 参数范围（斜率 -2000~2000）、性别一致性（DICOM vs 报告标注）、厂商名映射成功率
- **一致性：** DICOM 元数据与报告标注的交叉核对——性别是否一致、年龄偏差是否在容差内
- **可用性：** 有效信息病例占比——至少有 1 条疾病标注 + 1 条解剖部位的病例比例
- **术语标准化：** 标注术语模糊匹配 RadLex 标准词表（71 条，中英双语），将同义表述归一化
- **综合评分：** 四维评分的算术均值，阈值 0.80 判定 PASS/FAIL

**传统方式：** 治理完成后无系统化验证——抽查几个病例、凭经验判断质量。没有量化指标，无法向合作方（药企、多中心研究）证明数据可信度。

**我们的 AI 方式：** 自动化的四维量化评估 + LLM 语义抽查（检查规则引擎标注的假阳性/假阴性）+ AI 治理总结。形成从"治理"到\"证明治理到位"的完整证据链。

| 维度 | 传统方式 | RadClean |
|------|---------|----------|
| 验证方式 | 人工抽查，主观判断 | 四维量化 + AI 语义抽查 |
| 可复现性 | 低 | 规则部分 100% |
| 证据链 | 无 | 评分报告 + 语义审计 + AI 总结 |
| 术语标准化 | 手工映射 | 自动 RadLex 模糊匹配 |

## 三、平台与 AI 深度融合分析

### 3.1 架构设计：规则引擎为骨干，LLM 为增强

RadClean 的 AI 策略核心是**"规则优先，LLM 增强"**——不做端到端黑箱模型。

- **规则引擎负责确定性工作：** DICOM 字段清洗、厂商名映射、年龄格式转换等是确定性任务，规则引擎零错误率、零延迟、100% 可解释
- **LLM 负责语义理解：** 报告自由文本的结构化提取、长尾疾病表述识别、治理总结撰写是 LLM 的强项
- **两者不耦合：** LLM 不可用时规则引擎独立运行，不影响核心治理功能

### 3.2 AI 的三个应用层次

| 层次 | 功能 | AI 角色 |
|------|------|---------|
| L1 规则自动化 | 数据净化（Stage 1） | 确定性规则，无模型依赖 |
| L2 语义增强 | 报告结构化（Stage 2） | 规则引擎先行 + LLM 13 字段深度提取 |
| L3 智能审计 | 治理验证（Stage 3） | LLM 语义质量抽查 + AI 治理总结 |

### 3.3 AI 深度融合的四个特征

1. **非侵入式增强：** LLM 通过标准 API 调用（OpenAI 兼容协议），可切换不同模型而不影响管线主体
2. **实时流式交互：** Stage 2 的 LLM 分析结果以 SSE 流式推送至前端，用户可实时观察 AI 推理过程
3. **AI 自我审计：** Stage 3 的语义质量抽查让 LLM 审计规则引擎的标注质量——发现假阳性、假阴性、否定检测错误，形成 AI 辅助的质量闭环
4. **治理知识沉淀：** AI 治理总结将量化评分转化为人类可读的分析报告，降低治理结果的理解门槛

### 3.4 与传统纯规则引擎方案的本质区别

- 传统方案：规则清洗 → 输出结果 → 人工检查（开环）
- RadClean：规则清洗 → LLM 增强标注 → 四维量化评分 → LLM 语义审计 → AI 治理总结（闭环）

RadClean 不是"用 AI 替代规则"，而是"用 AI 让规则的产出更可信、更完整、更易用"。
